Moduł 01
Mental model:
Angular → React
Zanim napiszesz pierwszą linię Reacta — zrozum, co odpowiada czemu. To przyspieszy naukę 3×.
Fundamentalna różnica filozoficzna

Angular to framework — dostarcza gotowe rozwiązania na wszystko: DI, routing, formularze, HTTP, change detection. React to biblioteka UI — obsługuje tylko renderowanie. Resztę dobierasz sam.

To nie jest minus Reacta — to feature. Ekosystem się rozwinął i dla każdego problemu jest jedno dobre rozwiązanie, które robi robotę świetnie.

💡
Kluczowy mental shift: W Angularze pytasz "jak to zrobić w Angularze?". W Reakcie pytasz "jak to zrobić w JavaScript?" — React jest transparentny, JS jest na wierzchu.

Co zniknęło (i dlaczego to dobrze)
🗑
NgModule — W Reacte nie ma modułów. Każdy komponent to plik, który importujesz jak każdy inny moduł JS. Koniec z boilerplate declarations, imports, exports.
🗑
Zone.js i change detection — React nie ślędzi magicznie co się zmieniło. Ty wywołujesz setState → React wie że coś się zmieniło → re-render. Prostsze, przewidywalne.
🗑
RxJS jako domyślny tool — W Reacte nie ma Observables wszędzie. Async to async/await. RxJS możesz użyć jeśli chcesz, ale nie musisz.
🗑
Klasy komponentów — Żadnych class MyComponent implements OnInit. Wszystko to funkcje. React 16.8+ (2019) przestawił się na hooki — klasy wciąż działają, ale nikt ich nie pisze.
Co jest naprawdę nowe (od 2018)

W 2018 React był jeszcze klasowy. Oto co się zmieniło:

  • Hooks (React 16.8, 2019) — useState, useEffect i cała rodzina. To jest teraz React.
  • Concurrent Mode / Fiber (React 18, 2022) — renderowanie nie blokuje UI, Suspense, transitions.
  • useTransition, useDeferredValue (React 18) — kontrola priorytetu renderowania.
  • Server Components (React 19, 2024) — komponenty renderowane na serwerze, bez JS na kliencie.
  • use() hook (React 19) — await w komponencie, nowy sposób na dane asynchroniczne.
  • Actions / useActionState (React 19) — nowy model formularzy i mutacji.
✓
Na rozmowie wystarczy znać hooki dobrze, wiedzieć że React 18 wprowadził Concurrent Mode, i wiedzieć że React 19 to Server Components. Reszty nikt nie pyta na poziomie senior bez wcześniejszego ostrzeżenia.

Komponent to funkcja, która zwraca JSX

Tyle. To naprawdę tyle. Żadnych dekoratorów, żadnych klas, żadnego metadata.

TypeScript
Angular

  selector: 'app-hello',
  template: `
    <h1>Cześć, {{ name }}!</h1>
    <p>Wiek: {{ age }}</p>
  `
})

  @Input() name!: string;
  @Input() age!: number;

TypeScript + JSX
React
interface HelloProps {
  name: string;
  age: number;
}

export function Hello({ name, age }: HelloProps) {
  return (
    <div>
      <h1>Cześć, {name}!</h1>
      <p>Wiek: {age}</p>
    </div>
  );
}
💡
JSX to nie HTML. To JavaScript z syntaktycznym cukrem. <h1> kompiluje się do React.createElement('h1', null, ...). Dlatego: className zamiast class, htmlFor zamiast for, wszystkie atrybuty camelCase.
Renderowanie warunkowe

Nie ma *ngIf. Jest JavaScript:

JSX
React
function UserCard({ user, isLoading }: Props) {
  return (
    <div>
      {/* && — jak *ngIf */}
      {isLoading && <Spinner />}

      {/* ternary — jak *ngIf / else */}
      {user ? (
        <p>Witaj, {user.name}</p>
      ) : (
        <p>Zaloguj się</p>
      )}

      {/* switch — wyciągasz do zmiennej */}
      {renderStatus(user.status)}
    </div>
  );
}
⚠️
Pułapka: 0 && <Component /> renderuje 0 na ekranie! Zawsze używaj wyrażeń booleanowych: count > 0 && <Component /> lub !!count && <Component />.

& useEffect
Dwa najważniejsze hooki. Serce Reacta. Tu jest 80% tego co musisz wiedzieć.
useState — lokalny stan komponentu

Zamiast this.property = value masz setState. Zmiana stanu → React re-renderuje komponent.

TypeScript + JSX
React
import { useState } from 'react';

function Counter() {
  // [wartość, setter] — destrukturyzacja tablicy
  const [count, setCount] = useState(0);
  const [name, setName] = useState('');

  return (
    <div>
      <p>Licznik: {count}</p>
      <button onClick={() => setCount(count + 1)}>+</button>
      <button onClick={() => setCount(c => c - 1)}>-</button>
    </div>
  );
}

Functional update — kiedy używać

setCount(count + 1) — bezpośrednia wartość. OK gdy nie zależy na poprzednim stanie.

setCount(c => c + 1) — funkcja z poprzednim stanem. Zawsze używaj tego w callbackach asynchronicznych, bo count może być "stary" (closure problem).

⚠️
Stan jest immutable! Nigdy nie mutuj bezpośrednio:
❌ user.name = 'Anna' // React nie zobaczy zmiany!
✓ setUser({...user, name: 'Anna'})
useEffect — efekty uboczne i lifecycle

Jeden hook zastępuje ngOnInit, ngOnChanges, ngOnDestroy i ngAfterViewInit.

TypeScript
React
import { useEffect, useState } from 'react';

function UserProfile({ userId }: { userId: string }) {
  const [user, setUser] = useState(null);

  // 1. [] — działa raz, jak ngOnInit
  useEffect(() => {
    console.log('mounted!');
    return () => console.log('unmounted!'); // cleanup = ngOnDestroy
  }, []);

  // 2. [userId] — reaguje na zmianę userId, jak ngOnChanges
  useEffect(() => {
    fetch(`/api/users/${userId}`)
      .then(r => r.json())
      .then(data => setUser(data));
  }, [userId]); // dependency array!

  // 3. bez [] — działa po każdym renderze
  useEffect(() => {
    document.title = user?.name ?? 'Loading...';
  }); // ← brak tablicy = każdy render

  return <div>{user?.name}</div>;
}
⚠️
Najczęstszy błąd: dependency array. Jeśli używasz zmiennej wewnątrz useEffect, musi być w tablicy. ESLint plugin eslint-plugin-react-hooks to sprawdza — zawsze go używaj.
Async w useEffect — prawidłowy wzorzec
TypeScript
React
useEffect(() => {
  // useEffect NIE może być async — callback musi zwrócić cleanup lub nic
  // Rozwiązanie: IIFE lub osobna funkcja wewnątrz

  const fetchUser = async () => {
    try {
      const data = await getUser(userId);
      setUser(data);
    } catch (err) {
      setError(err);
    }
  };

  fetchUser();

  // Cleanup — anuluj request jeśli komponent się odmontuje
  const controller = new AbortController();
  return () => controller.abort();
}, [userId]);
✓
Tip: W praktyce do fetchowania danych nie używaj useEffect + useState bezpośrednio. Użyj TanStack Query (Moduł 8) — obsługuje loading, error, cache, deduplication za Ciebie.

useRef — bez re-renderu

useRef to kontener, który nie powoduje re-renderu przy zmianie. Używasz go do: dostępu do DOM, zapamiętania wartości między renderami (jak poprzednia wartość), timerów/eventListenerów.

TypeScript
React
function SearchInput() {
  // Dostęp do DOM — jak ViewChild w Angular
  const inputRef = useRef<HTMLInputElement>(null);

  // Wartość między renderami bez trigger re-render
  const timerRef = useRef<ReturnType<typeof setTimeout>>();

  const focusInput = () => {
    inputRef.current?.focus(); // optional chaining!
  };

  const handleChange = (e: ChangeEvent<HTMLInputElement>) => {
    clearTimeout(timerRef.current);
    timerRef.current = setTimeout(() => {
      search(e.target.value);
    }, 300);
  };

  return <input ref={inputRef} onChange={handleChange} />;
}
useMemo i useCallback — optymalizacja
💡
Zasada: Nie używaj ich domyślnie. Używaj gdy: 1) masz pomierzony problem z wydajnością, 2) przekazujesz callback do memoizowanego child-komponentu, 3) wyliczasz coś kosztownego na każdym renderze.
TypeScript
React
function ProductList({ products, filters }) {
  // useMemo — memoizuje wynik obliczeń
  // Jak pure pipe w Angular — nie przelicza bez potrzeby
  const filteredProducts = useMemo(() => {
    return products.filter(p =>
      p.category === filters.category &&
      p.price <= filters.maxPrice
    );
  }, [products, filters]); // przelicz tylko gdy się zmieniły

  // useCallback — memoizuje funkcję
  // Bez tego: nowa referencja na każdym renderze → child się re-renderuje
  const handleAddToCart = useCallback((productId: string) => {
    addToCart(productId, filters.userId);
  }, [filters.userId]);

  return (
    <List
      items={filteredProducts}
      onAdd={handleAddToCart} // stable reference!
    />
  );
}
useContext — lekki DI

Odpowiednik Dependency Injection z Angulara, ale prostszy. Używaj do: theme, auth state, language, user preferences. Nie do wszystkiego — do tego jest Zustand (Moduł 7).

TypeScript
React
// 1. Stwórz context
interface AuthContext {
  user: User | null;
  logout: () => void;
}
const AuthContext = createContext<AuthContext>(null!);

// 2. Provider — otocz aplikację lub drzewo
export function AuthProvider({ children }) {
  const [user, setUser] = useState<User | null>(null);
  const logout = useCallback(() => setUser(null), []);
  return (
    <AuthContext.Provider value={{ user, logout }}>
      {children}
    </AuthContext.Provider>
  );
}

// 3. Custom hook — ergonomiczne użycie
export function useAuth() {
  const ctx = useContext(AuthContext);
  if (!ctx) throw new Error('useAuth musi być w AuthProvider');
  return ctx;
}

// 4. Użycie w dowolnym komponencie
function Header() {
  const { user, logout } = useAuth();
  return <button onClick={logout}>Wyloguj {user?.name}</button>;
}
Custom hooks — odpowiednik serwisów

Custom hook to funkcja zaczynająca się od use, która może używać innych hooków. To jest wzorzec kompozycji w React — wyciągasz logikę do reużywalnej funkcji.

TypeScript
Custom Hook
// useLocalStorage — persystentny stan
function useLocalStorage<T>(key: string, initial: T) {
  const [value, setValue] = useState<T>(() => {
    const stored = localStorage.getItem(key);
    return stored ? JSON.parse(stored) : initial;
  });

  const set = useCallback((val: T) => {
    setValue(val);
    localStorage.setItem(key, JSON.stringify(val));
  }, [key]);

  return [value, set] as const;
}

// useDebounce — debounce wartości
function useDebounce<T>(value: T, delay: number) {
  const [debounced, setDebounced] = useState(value);
  useEffect(() => {
    const timer = setTimeout(() => setDebounced(value), delay);
    return () => clearTimeout(timer);
  }, [value, delay]);
  return debounced;
}

// Użycie
function SearchPage() {
  const [query, setQuery] = useLocalStorage('search', '');
  const debouncedQuery = useDebounce(query, 300);
  // ...
}

Moduł 05
Formularze
& React Hook Form
Formularze w React to jeden z najważniejszych tematów. React Hook Form to de facto standard — lżejszy i prostszy niż Reactive Forms.
Controlled vs Uncontrolled

W Angular wszystko było "controlled" przez FormControl. React ma dwa podejścia:

JSX
Controlled input
// Stan jest w React — React kontroluje
const [email, setEmail] = useState('');

<input
  value={email}
  onChange={e => setEmail(e.target.value)}
/>

// Każde naciśnięcie klawisza → re-render
// Dobre do: live validation, dependent fields
JSX
Uncontrolled (ref)
// DOM przechowuje wartość — React nie kontroluje
const emailRef = useRef();

<input ref={emailRef} />

// Wartość przy submit:
const val = emailRef.current.value;

// Dobre do: proste formy, duże formularze
// React Hook Form używa tego podejścia!
React Hook Form — standard w 2025

Działa uncontrolled (zero re-renderów przy pisaniu), ma integrację z TypeScript i Zod, obsługuje złożone walidacje.

TypeScript
React Hook Form + Zod
import { useForm } from 'react-hook-form';
import { zodResolver } from '@hookform/resolvers/zod';
import { z } from 'zod';

// 1. Schema walidacji
const schema = z.object({
  email: z.string().email('Nieprawidłowy email'),
  password: z.string().min(8, 'Min. 8 znaków'),
  age: z.number().min(18),
});

type FormData = z.infer<typeof schema>;

function LoginForm() {
  const {
    register,      // rejestruje pole
    handleSubmit,  // wrapper na submit
    formState: { errors, isSubmitting },
    watch,         // obserwuje wartość pola
    setValue,      // programowe ustawianie
  } = useForm<FormData>({
    resolver: zodResolver(schema),
    defaultValues: { email: '', password: '' },
  });

  const onSubmit = async (data: FormData) => {
    await login(data); // data jest już zwalidowana i typed
  };

  return (
    <form onSubmit={handleSubmit(onSubmit)}>
      <input
        {..register('email')}
        placeholder="Email"
      />
      {errors.email && (
        <span>{errors.email.message}</span>
      )}

      <input
        {..register('password')}
        type="password"
      />
      {errors.password && (
        <span>{errors.password.message}</span>
      )}

      <button disabled={isSubmitting}>
        {isSubmitting ? 'Logowanie...' : 'Zaloguj'}
      </button>
    </form>
  );
}
✓
Porównanie z Angular Reactive Forms: brak boilerplate FormGroup/FormControl, typy wyciągane z Zod schema automatycznie, zero re-renderów przy pisaniu (uncontrolled), integracja z dowolnym komponentem UI przez Controller.

Moduł 07
Global state —
Zustand
Zarządzanie stanem globalnym bez Redux-owego boilerplate. Zustand to 1KB biblioteki, która robi wszystko czego potrzebujesz.
Dlaczego nie Redux?

Redux był standardem przez lata, ale Redux Toolkit (RTK) wciąż wymaga sporo konfiguracji: actions, reducers, selectors, configureStore. Zustand to całkowite przeciwieństwo — minimal API, zero boilerplate.

📊
Kiedy co: Zustand → 90% projektów. Redux Toolkit → gdy masz skomplikowany flow z middleware, lub pracujesz z legacy kodem. Jotai/Recoil → atomic state. Context API → mało zmieniające się dane (theme, locale).
Tworzenie store
TypeScript
Zustand store
import { create } from 'zustand';
import { persist } from 'zustand/middleware';

interface CartStore {
  items: CartItem[];
  total: number;
  addItem: (item: CartItem) => void;
  removeItem: (id: string) => void;
  clearCart: () => void;
}

export const useCartStore = create<CartStore>()(
  persist(  // middleware: auto-zapisuje do localStorage!
    (set, get) => ({
      items: [],
      total: 0,

      addItem: (item) =>
        set(state => {
          const existing = state.items.find(i => i.id === item.id);
          if (existing) {
            return {
              items: state.items.map(i =>
                i.id === item.id
                  ? {...i, qty: i.qty + 1}
                  : i
              )
            };
          }
          return { items: [...state.items, {...item, qty: 1}] };
        }),

      removeItem: (id) =>
        set(state => ({
          items: state.items.filter(i => i.id !== id)
        })),

      clearCart: () => set({ items: [], total: 0 }),
    }),
    { name: 'cart-storage' }
  )
);

function UserList() {
  const {
    data: users,
    isLoading,
    isError,
    error,
    refetch,
  } = useQuery({
    queryKey: ['users'],       // unikalny klucz cache

    staleTime: 5 * 60_000,      // 5 minut cache

  if (isError) return <Error message={error.message} />;

  return (
    <ul>
      {users.map(u => (
        <li key={u.id}>{u.name}</li>
      ))}
    </ul>
  );
}

function CreateUserForm() {
  const queryClient = useQueryClient();

  const mutation = useMutation({
    mutationFn: (newUser: CreateUserDto) =>
      createUser(newUser),

    onSuccess: () => {
      // Invalidate → automatyczny refetch listy użytkowników

      toast.error(`Błąd: ${error.message}`);
    },
  });

    <form onSubmit={handleSubmit(data => mutation.mutate(data))}>
      {/* ... pola formularza */}
      <button
        type="submit"
        disabled={mutation.isPending}
      >
        {mutation.isPending ? 'Zapisywanie...' : 'Zapisz'}
      </button>
    </form>
  );
}

TypeScript
React
const ModalContext = createContext<{ onClose: () => void }>(null!);

function Modal({ children, onClose }) {
  return (
    <ModalContext.Provider value={{ onClose }}>
      <div className="modal-overlay" onClick={onClose}>
        <div className="modal" onClick={e => e.stopPropagation()}>
          {children}
        </div>
      </div>
    </ModalContext.Provider>
  );
}

function ModalHeader({ children }) {
  const { onClose } = useContext(ModalContext);
  return (
    <div className="modal-header">
      {children}
      <button onClick={onClose}>✕</button>
    </div>
  );
}

// Attach sub-komponenty do Modal
Modal.Header = ModalHeader;
Modal.Body = ({ children }) => <div className="modal-body">{children}</div>;
Modal.Footer = ({ children }) => <div className="modal-footer">{children}</div>;
Kluczowe zasady architektury
  • Single Responsibility — każdy komponent robi jedną rzecz. Logika biznesowa → custom hooks, UI → komponenty.
  • Colocate — trzymaj pliki blisko tego gdzie są używane. Nie jeden globalny folder "components".
  • Prezentacyjne vs kontenerowe — komponenty UI nie wiedzą skąd dane. Wyżej w drzewie: fetching i logika.
  • Feature-based struktura folderów — features/users/, features/cart/ zamiast components/, services/.
  • Barrel exports — index.ts w każdym module, żeby importować from 'features/users'.

Moduł 10
Testy —
React Testing Library
Testuj zachowanie, nie implementację. RTL i Vitest to standard w 2025.
Filozofia RTL — testuj jak użytkownik

React Testing Library ma jedno motto: "The more your tests resemble the way your software is used, the more confidence they can give you."

💡
Mapowanie z Angular Testing: TestBed.createComponent → render(). fixture.nativeElement.querySelector → screen.getByRole/Text/Label. fixture.detectChanges() → fireEvent / userEvent.
Priorytety selektorów (od najlepszego)
  • getByRole — semantyczne (button, heading, textbox) — zawsze preferuj
  • getByLabelText — pola formularza przez label
  • getByPlaceholderText — przez placeholder (słabszy)
  • getByText — tekst widoczny dla użytkownika
  • getByTestId — data-testid — tylko jako ostateczność
Przykładowy test
TypeScript
RTL + Vitest
import { render, screen, waitFor } from '@testing-library/react';
import userEvent from '@testing-library/user-event';
import { describe, it, expect, vi } from 'vitest';
import { LoginForm } from './LoginForm';

describe('LoginForm', () => {
  it('pokazuje błąd przy nieprawidłowym email', async () => {
    const user = userEvent.setup();
    render(<LoginForm />);

    // Znajdź input przez role + name (label)
    const emailInput = screen.getByRole('textbox', {
      name: /email/i
    });

    // Symuluj wpisywanie (userEvent > fireEvent)
    await user.type(emailInput, 'nieemail');
    await user.click(screen.getByRole('button', { name: /zaloguj/i }));

    // Sprawdź czy pojawił się błąd
    await waitFor(() => {
      expect(screen.getByText(/nieprawidłowy email/i)).toBeInTheDocument();
    });
  });

  it('wywołuje onLogin z danymi po udanym submit', async () => {
    const onLogin = vi.fn();
    const user = userEvent.setup();
    render(<LoginForm onLogin={onLogin} />);

    await user.type(screen.getByRole('textbox', {name: /email/i}), 'a@b.com');
    await user.type(screen.getByLabelText(/password/i), 'Pass1234!');
    await user.click(screen.getByRole('button', {name: /zaloguj/i}));

    expect(onLogin).toHaveBeenCalledWith({
      email: 'a@b.com',
      password: 'Pass1234!',
    });
  });
});
